"Tornike, what do you think of AI coding tools?"

I like to answer this frequent question by way of a short parable. A parable of two companies. It goes something like this:

Imagine two very similar companies. Both companies generate similar revenue and produce a similar SaaS product. The only difference between these companies is that Company A has 1 million lines of code and Company B only has 100 thousand lines of code. Which company is better?

Clearly, the company with fewer lines of code is better. They have fewer lines of code to maintain and so they can iterate more quickly. All other things being equal, less code is better. This means that code is a form of debt. If you use an AI to generate code, you are effectively getting a debt -- a debt of code.

Is it worth going into code debt? It depends. Debt can be both good or bad. If used correctly, debt can allow faster growth. Used incorrectly, it can [implode](https://www.bbc.com/news/articles/ce87rer52k3o). What is important is having access to these debt-generating tools. They can allow faster growth.

"Will AI replace human programmers?"

I don't think so. Did [domestication of the horse](https://en.wikipedia.org/wiki/Domestication_of_the_horse) replace walking? No, but it instead replaced only one kind of walking with long distances and heavy loads. Did the invention of the automobile replace horses? Mostly yes! Still, walking remains irreplaceable. I think this is how AI coding tools will behave. As a car they will augment the programmer and allow them to cover the most boring stretches of programming. The need for programming and programmers will remain still.
